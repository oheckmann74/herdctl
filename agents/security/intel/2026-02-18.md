## New Findings

### Finding #011: sessions.ts shell:true with unvalidated session_id

**Severity:** MEDIUM-LOW
**Location:** `packages/cli/src/commands/sessions.ts:412-416`
**Discovered:** 2026-02-18

**Code:**
```typescript
const claudeArgs = ["--resume", session.session_id];
const child = spawn("claude", claudeArgs, {
  cwd,
  stdio: "inherit",
  shell: true,  // ← unnecessary shell mode
});
```

**Schema:**
```typescript
// session-info.ts:42 - only min(1), no character restrictions
session_id: z.string().min(1, "Session ID cannot be empty"),
```

**Issue:** With `shell: true`, Node.js converts the argument array into a shell command string. If `session_id` contains shell metacharacters (`; & | $ \` !`), they would be interpreted by the shell. The `session_id` schema (`z.string().min(1)`) does not restrict characters.

**Attack path:** An attacker who can write to `.herdctl/sessions/<agent-name>.json` could craft a `session_id` containing `; malicious-command` which would execute when the operator runs `herdctl sessions resume`.

**Severity rationale (MEDIUM-LOW, not HIGH):**
- Requires write access to `.herdctl/sessions/` - same trust level as operator
- CLI tool used interactively by trusted fleet operators
- Session files not network-accessible; not writable by Docker agents
- Session IDs in practice are UUIDs generated by Claude CLI
- Fix is trivial: remove `shell: true` (not needed since `spawn` with an array is inherently safe)

**Recommended fix:**
```typescript
// Remove shell: true - spawn with array args doesn't need shell interpretation
const child = spawn("claude", claudeArgs, {
  cwd,
  stdio: "inherit",
  // shell: true  ← remove this
});
```

Or, add UUID pattern validation to `SessionInfoSchema`:
```typescript
session_id: z.string().regex(/^[a-zA-Z0-9_-]+$/, "Session ID must contain only safe characters"),
```
